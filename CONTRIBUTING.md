# Contributing to the Nexus Exchange API spec

This repository is the **source of truth** for the Nexus Exchange API. The
artifact is [`openapi.json`](./openapi.json) — a single OpenAPI 3 document
describing every REST and WebSocket endpoint. There is no SDK or runnable code
here; the spec itself is what we ship, validated by CI and released as a
versioned, downloadable asset.

If you found an inaccuracy or want to propose a change, thank you — this guide
explains how.

## What lives here

- `openapi.json` — the spec. The only file that defines the API surface.
- `CHANGELOG.md` — generated by release-please. **Do not hand-edit.**
- `redocly.yaml` — lint config for the spec.
- `.release-please-manifest.json` / `release-please-config.json` — release
  automation config.

## Proposing a change

### 1. Edit `openapi.json`

Make your change directly in `openapi.json`. Keep it valid OpenAPI 3 and make
sure every operation has a unique `operationId` — code generators downstream
depend on those being present and stable.

### 2. Validate locally

CI runs [Redocly](https://redocly.com/docs/cli/) to lock in structural
validity and the `operationId` guarantees. Run the exact same check before you
push:

```bash
npx -y @redocly/cli@2 lint openapi.json
```

To preview how your change classifies against `main` — additive versus
breaking — run [`oasdiff`](https://github.com/oasdiff/oasdiff) the same way CI
does:

```bash
# Save the current main spec, then compare your working copy against it.
git show origin/main:openapi.json > /tmp/base.json

# Human-readable summary of every change.
oasdiff changelog /tmp/base.json openapi.json

# Breaking changes only (this is the gate CI enforces).
oasdiff breaking /tmp/base.json openapi.json --fail-on ERR
```

### 3. Write conventional-commit PRs

Versioning and the changelog are fully automated by
[release-please](https://github.com/googleapis/release-please) — the
**conventional commit type carries the intent**. Use these prefixes in your
commit messages (and PR title):

| Commit | Example | Effect |
|---|---|---|
| `feat:` | `feat: add GET /funding/history endpoint` | Additive change → patch bump |
| `fix:` | `fix: correct quantity type on POST /orders` | Backwards-compatible fix → patch bump |
| `feat!:` | `feat!: remove deprecated /v0/orders endpoint` | Breaking change → minor bump |
| `docs:` / `chore:` / `ci:` | `docs: clarify auth description` | No release |

A breaking change is signalled either by the `!` (e.g. `feat!:`) or by a
`BREAKING CHANGE:` footer in the commit body. Both bump the minor version.

You do **not** edit the version yourself. Spec PRs land on `main` without a
version bump; release-please accumulates merged commits into a running release
PR. Merging that release PR is the deliberate, batched release — it bumps
`$.info.version` inside `openapi.json`, updates `CHANGELOG.md`, tags `vX.Y.Z`,
cuts a GitHub Release, and attaches `openapi.json` as a release asset.

## Versioning and the breaking-change policy

The spec is pre-1.0 semver, as described in the README:

| Version bump | Meaning |
|---|---|
| `0.X.0` | Breaking change — removed endpoint, renamed field, new required parameter |
| `0.X.Y` | New feature or backwards-compatible fix |

Pre-1.0, a breaking change is *allowed* — but it must be **deliberate**, not a
surprise. The policy:

- **Prefer additive, optional changes.** New endpoints, new optional fields,
  and new optional parameters are non-breaking and ship as patch releases.
- **Make renames additive.** Instead of renaming a field, add the new field,
  keep the old one, and mark the old one `deprecated: true`. Remove it in a
  later, intentional breaking release.
- **Batch and minimize breaks.** Each breaking change ripples downstream (see
  below). When a break is genuinely necessary, group related breaks into one
  minor release rather than spreading them across several.

### How CI handles breaking changes

The **API diff** workflow runs `oasdiff` on every PR that touches
`openapi.json`. It posts an informational changelog and then runs:

```bash
oasdiff breaking /tmp/base.json openapi.json --fail-on ERR
```

If that finds a breaking change, the PR is **blocked** until you acknowledge it
deliberately:

1. Use a `feat!:` commit (or a `BREAKING CHANGE:` footer) so release-please
   bumps the minor version, and
2. Add the `breaking-change` label to the PR to acknowledge and unblock it.

This makes every break a conscious, two-step decision.

## Downstream ripple

Released spec versions are consumed by the downstream SDKs — Rust (`-rs`),
Python (`-py`), and the MCP server (`-mcp`) — which **pin to a released spec
version**. Every breaking change forces each of those to bump their pinned
version and regenerate. Be deliberate: a one-line rename here is a coordinated
release across several repos.

## Toward 1.0

While pre-1.0, breaking changes are permitted under the rules above. As the API
stabilizes toward `1.0.0`, the bar for breaking changes rises sharply — at and
after 1.0, breaks require a major version bump and a migration path. Designing
changes additively now keeps that transition cheap, so favor optional,
deprecation-first changes wherever you can.
